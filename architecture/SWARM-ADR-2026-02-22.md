## AGENTE 1 — ARQUITECTO DE SISTEMAS

**Pregunta central:** *"Es esta la arquitectura tecnica correcta para escalar de 5 a 50 clientes?"*

**Veredicto:** La arquitectura es viable para 5 clientes con intervencion manual frecuente. A 50 clientes, va a fallar en al menos 4 puntos criticos si no se implementan mitigaciones antes de llegar a ~15 clientes.

### 1.1 GHL Multi-tenant: Un numero WhatsApp para todos

**Rate Limiting (CRITICA):**
- 50 clientes en modo diario = 165-235 mensajes/dia en un solo numero
- GHL tiene throttling rules no documentadas (~30 msg/min en cuentas estandar)
- Fix MVP: Buffer de envio con delay de 2s entre mensajes, nunca mas de 1 msg/seg

**Confusion de Routing (Alta):**
- `phone_owner TEXT UNIQUE NOT NULL` = un solo telefono por restaurante
- Que pasa si el dueno cambia de telefono? Si su esposa escribe? Si tiene dos numeros?
- Fix MVP: Tabla `authorized_contacts` con multiples telefonos por restaurante

**Sin Aislamiento de Fallos (Alta):**
- Error en sub-account de un cliente puede bloquear procesamiento de TODOS
- Fix MVP: try-catch aislado por cliente en cada webhook

### 1.2 Webhook Routing: Problemas criticos

**Webhooks Duplicados (CRITICA):**
- GHL reintenta webhooks si no recibe HTTP 200 en <5 seg
- Si Claude tarda 2-5s clasificando intent, GHL reenvia = mensajes duplicados
- El PRD NO menciona idempotencia en ningun lado
- Fix MVP OBLIGATORIO:
  1. Responder HTTP 200 INMEDIATAMENTE (<100ms)
  2. Procesar async
  3. Campo `ghl_event_id TEXT UNIQUE` en whatsapp_log para idempotencia

**Sin Dead Letter Queue (Media):**
- Si el procesamiento falla, el mensaje se pierde para siempre
- Fix MVP: Tabla `webhook_raw_log` que guarda payload raw ANTES de procesar

**Sin Verificacion de Firma (Media):**
- `GHL_WEBHOOK_SECRET` existe en .env pero no hay validacion en el handler
- Fix MVP: Validar header de firma en cada request

### 1.3 Supabase Schema: Problemas de diseno

**Indices Faltantes (Media a 50 clientes):**
```sql
CREATE INDEX idx_content_restaurant_status ON content_items(restaurant_id, status);
CREATE INDEX idx_content_scheduled ON content_items(scheduled_at) WHERE status = 'approved';
CREATE INDEX idx_whatsapp_restaurant ON whatsapp_log(restaurant_id, created_at DESC);
CREATE INDEX idx_leads_stage ON leads(stage);
CREATE INDEX idx_restaurants_status ON restaurants(status);
```

**Falta relacion content_items <-> whatsapp_log (Media):**
- Cuando el cliente responde "checkmark", como sabe el sistema CUAL post esta aprobando?
- En modo semanal con 6 posts, esto es CRITICO
- Fix: `content_item_id UUID REFERENCES content_items(id)` en whatsapp_log

**Sin Tabla de Subscripciones (Media):**
- Solo `tier TEXT DEFAULT '$99'` en restaurants. No hay tracking de pagos
- Fix: Tabla `subscriptions` con stripe_customer_id, status, period_start/end

### 1.4 node-cron: El talon de Aquiles

**No Sobrevive Reinicios (CRITICA):**
- Si el servidor crashea a las 7:50am, los cron de las 8:00am nunca se ejecutan
- NADIE se entera hasta que un cliente pregunta "donde esta mi post?"
- Fix MVP OBLIGATORIO: Tabla `scheduled_jobs` con persistencia en DB. node-cron solo actua como poller cada minuto buscando jobs pendientes

**Timezones (Media):**
- "8am" de quien? Charlotte (EST) vs Los Angeles (PST) = 3 horas
- Fix: Campo `timezone TEXT DEFAULT 'America/New_York'` en restaurants

### 1.5 Claude API en ruta critica

**Latencia de Clasificacion (Alta):**
- Cliente envia "checkmark", espera confirmacion RAPIDA
- Claude tarda 2-5 seg en clasificar = cliente piensa que no funciono = escribe de nuevo = duplicado
- Fix MVP: Clasificador regex ANTES de Claude para intents simples (aprobacion, rechazo, escalacion)
- Reduce llamadas a Claude en ~40-60% y baja latencia a <100ms para lo mas frecuente

**Sin Fallback (Media):**
- Si Claude API cae, el bot queda 100% muerto
- Fix: Regex como fallback para aprobaciones + respuestas pre-escritas para FAQs

### 1.6 Sin Sistema de Colas

- Todo se procesa sincronicamente en el request del webhook
- Fix MVP sin Redis: "Database as Queue" pattern — webhook guarda payload, worker loop cada 5s procesa pendientes

### 1.7 Sin Conversation State Machine

- Multiples flujos conversacionales (onboarding, tripwire, emergencia, aprobacion, FAQ)
- Si el cliente esta a mitad de un flujo y cambia de tema, se pierde el contexto
- Fix: Tabla `conversation_state` con current_flow, flow_step, flow_data, expires_at

### Donde VA a fallar en produccion (en orden cronologico):

1. **Dia 1-7:** Webhook con formato inesperado crashea Node.js. Todos sin servicio.
2. **Dia 7-14:** Server se reinicia, cron pierde estado, nadie recibe contenido matutino.
3. **Dia 14-30:** Modo semanal: 6 posts, cliente dice "checkmark" — sistema no sabe a cual.
4. **Cliente 10-15:** Rate limits de GHL a las 8am. Mensajes perdidos/duplicados.
5. **Cliente 20+:** Human review de 20+ posts diarios se vuelve cuello de botella operativo.

## AGENTE 2 — PRODUCT SKEPTIC

**Pregunta central:** *"El producto resuelve el dolor correcto? El funnel va a convertir?"*

### 2.1 El dolor central: Estamos resolviendo un sintoma

Un dueno de taqueria NO se despierta pensando "necesito mas contenido para Instagram". Se despierta pensando:
- "Por que el viernes no vino nadie?" (foot traffic)
- "Tengo 3 resenas de 1 estrella en Google" (reputacion)
- "El rent me subio" (supervivencia financiera)

**Contenido para redes es un problema de nivel 3-4 en su jerarquia.** Estamos vendiendo una vitamina, no una aspirina.

**Recomendacion:** El pitch NO puede ser "te hacemos contenido". Tiene que ser "te traemos mas clientes". El contenido es el medio, no el fin.

### 2.2 El funnel MIFGE -> Tripwire -> Bridge -> $99

**MIFGE (AI Photoshoot gratis):**
- Un extrano manda fotos "mejoradas" por WhatsApp = "quien es este? es una estafa?"
- La comunidad latina es especialmente cautelosa con contactos no solicitados
- El "uncanny valley gastronomico": fotos bonitas pero falsas pierde autenticidad
- PREGUNTA CLAVE: Se ha probado esto con UNA persona real?

**Tripwire $19.99:**
- Tierra de nadie: muy barato para parecer profesional, muy caro para impulso
- Compite con "mi sobrina me hace eso en Canva gratis"
- Cobrar por WhatsApp es friccional. Muchos operan en cash.

**Bridge (7 dias):**
- 7 dias es una eternidad para un restaurantero ocupado
- "Free week of content" — operacionalmente, como le diste acceso a sus redes en 1 semana?
- Nurturing por email NO funciona en este segmento

**El funnel completo tiene DEMASIADOS PASOS:**
- Minimo 12-15 touchpoints para un ticket de $99/mes
- Conversion estimada end-to-end: 1-2%
- Para 3-5 clientes necesitas enviar 200-500 MIFGEs

### 2.3 Objeciones del restaurantero

| Objecion | Realidad | Como contrarrestar |
|----------|----------|-------------------|
| "No tengo tiempo" | Incluso el checkmark toma 2-3 min que NO tiene | Autopilot por defecto, solo notificar |
| "Mi sobrino me maneja las redes" | El competidor REAL, no Owner.com. Gratis vs $99 | "Tu sobrino publica cuando puede, nosotros TODOS LOS DIAS" |
| "Ya probe algo asi y no funciono" | Cicatriz emocional real | Garantia de devolucion + resultados rapidos |
| "No veo resultados del social media" | MATA el negocio si no se resuelve | Cupones trackables + mini-reporte mensual |
| "No confio en AI con mi marca" | Legitima. Un error cultural destruye confianza | Human-in-the-loop + validacion cultural |

### 2.4 Pricing

**Para el cliente:**
- Taqueria promedio: $250K-$400K revenue/ano, margen neto 3-9%
- $99/mes = entre 3% y 16% de ganancia neta
- Para un restaurante que gana $7,500 netos/ano, $1,188 es 16% de su ganancia

**Para Marketero:**
- Costos por cliente/mes: Claude $5-15 + fal.ai $10-20 + GHL $5-10 + WhatsApp $5-10
- CON supervision humana significativa: margen NEGATIVO a $99
- Break-even optimista: ~7 clientes. Realista: ~20 clientes.
- Primeros 2 meses seran a perdida. Normal para startup pero hay que saberlo.

### 2.5 Confianza desde el dia 1

- No hay portfolio, testimonios, ni presencia propia
- La comunidad latina funciona con confianza personal (compadre, iglesia, mercado)
- Un mensaje frio de WhatsApp tiene probabilidad altisima de ser ignorado
- **Los primeros 5-10 clientes REQUIEREN visita en persona**

### 2.6 Riesgos de retencion

- "AI-ish look" despues del mes 2-3: fatiga de contenido
- Sin mecanismo de atribucion de ROI = churn masivo
- Estacionalidad: enero-febrero el dueno corta gastos, $99 es el primer corte
- Churn proyectado sin ROI proof: 15-25% mensual

### Fortalezas reales del producto:
1. Nicho bien definido: latino + espanol + WhatsApp = nadie mas lo hace
2. WhatsApp como canal de delivery es brillante
3. $99/mes es accesible para cualquier restaurante
4. Los 2 pilotos confirmados son oro puro
5. TAM de 67,000 restaurantes es suficiente

## AGENTE 3 — UX / WHATSAPP EXPERIENCE DESIGNER

**Pregunta central:** *"Es la experiencia WhatsApp intuitiva para un dueno de 50 anos con iPhone basico?"*

**Veredicto: NO todavia.** El diseno esta pensado desde la perspectiva del sistema, no del usuario.

### 3.1 El emoji checkmark como aprobacion es una bomba de tiempo

El cliente va a enviar: "si", "ok", "dale", "va", "sale", "esta bien", thumbs up emoji (con tono de piel), reaccion nativa de WhatsApp (que NO es un mensaje). El sistema solo reconoce checkmark.

**Fix OBLIGATORIO:** Reconocer 20+ variantes de "si" en espanol + 6 variantes de thumbs up + checkmark. Siempre confirmar: "Listo! Tu post se publicara hoy a las 2pm". Si no entiende: "No entendi bien. Quieres que publique este post? Responde SI o NO."

### 3.2 Sin mecanismo de "deshacer"

Maria toca sin querer el boton de enviar con las manos con grasa. Se publica un post con precio incorrecto. No hay reversion.

**Fix:** Buffer de 15 minutos: "Post aprobado. Se publicara en 15 minutos. Responde CANCELAR si quieres detenerlo."

### 3.3 Timer de autopilot invisible

2 horas de autopilot sin que el cliente vea ningun timer ni reciba recordatorio.

**Fix:** Recordatorio a los 90 minutos: "Tu post se publicara automaticamente en 30 minutos. Si quieres hacer cambios, responde ahora."

### 3.4 Modo semanal = CAOS

**6 posts el lunes = 12-18 mensajes de WhatsApp de golpe.** Se siente como spam. El cliente tiene que scrollear. Si responde "checkmark", a cual post se refiere?

**Aprobacion parcial imposible de parsear:**
"Los primeros 3 estan bien, pero el de viernes cambialo y el de sabado usa otra foto"
= una sola frase con aprobacion de 3, cambio de texto en 1, cambio de imagen en otro, y 1 sin mencionar.

**Fix RECOMENDADO:** Eliminar modo semanal en MVP. Enviar UN post a la vez, la noche anterior. Cada interaccion es simple: un post, si o no? Sin confusion posible.

**Fix alternativo (si se mantiene semanal):** Posts numerados 1-6 con dia asignado. Responder "TODOS" para aprobar todos, o el numero del post para cambiar.

### 3.5 Onboarding de 60 minutos es IRREAL

El usuario:
- Trabaja 12-16 horas al dia
- Manos con grasa/harina
- Constantemente interrumpido
- iPhone viejo con poca memoria
- Fotos mezcladas con fotos familiares

**Lo que va a pasar:** A los 15 minutos dice "espera, me habla un proveedor". Regresa 20 minutos despues sin recordar donde estaban.

**Fix:** Micro-sesiones de 5 minutos en 3-4 dias:
- Sesion 1 (5 min): Nombre, direccion, foto del frente
- Sesion 2 (5 min, dia siguiente): 5 fotos de platillos
- Sesion 3 (5 min): Logo (o generamos uno)
- Sesion 4 (10 min): Mostrar fotos mejoradas + primer post
- Total: 25 min distribuidos. El cliente nunca siente la carga.

**Eliminar preguntas tecnicas:**
- NO preguntar "cuales son tus colores de marca?"
- SI preguntar "de que color es el letrero de tu restaurante?" o extraer colores de la foto automaticamente
- NO preguntar "cual es tu tono de comunicacion?"
- SI observar como escribe el dueno y adaptar

### 3.6 NOTAS DE VOZ: El elefante en la habitacion

**Los usuarios latinos envian 3x mas notas de voz que texto.** El PRD no menciona soporte para notas de voz NI UNA VEZ.

Maria va a enviar una nota de voz de 45 segundos con: aprobacion condicional + cambio de precio + info nueva del restaurante + solicitud futura. Si el bot no procesa notas de voz, Maria no entiende por que no le responden.

**Fix OBLIGATORIO:** Whisper API para transcripcion. Costo: ~$0.006/minuto (~$0.003 por nota promedio de 30s). Es insignificante. **Esto NO es opcional para el mercado latino.**

### 3.7 Sobrecarga de mensajes

Calculo semanal: posts para aprobar (~12-18 msgs) + confirmaciones (6) + reporte (3-5) + recordatorios (2-6) = **23-35 mensajes semanales de un solo servicio en WhatsApp.**

**Fix:**
1. Ventana de envio 9am-8pm configurable
2. Agrupar mensajes (1 resumen en lugar de 6 confirmaciones)
3. Confirmaciones silenciosas: 1 resumen semanal en vez de 6 confirmaciones individuales

### 3.8 Estados de error no planificados

| Fallo | Lo que el cliente ve | Lo que deberia ver |
|-------|---------------------|-------------------|
| Imagen no se genera | Silencio | "Problema tecnico. Lista en max 1 hora" |
| GHL no publica | Cree que publico, no publico | "Hubo un problema, equipo trabajando" |
| Bot no entiende | Respuesta generica | "No entendi. Quieres hablar con una persona?" |
| WhatsApp se cae | Nada funciona | Plan B: publicar con autopilot, notificar despues |

### Las 3 decisiones de diseno que transforman la experiencia:
1. **Procesar notas de voz** — diferencia entre producto que entiende a su usuario vs que lo obliga a adaptarse
2. **Un post a la vez** — elimina toda la complejidad del modo semanal
3. **Confirmar todo, asumir nada** — el cliente nunca esta en incertidumbre

## AGENTE 4 — GROWTH & GTM STRATEGIST

**Pregunta central:** *"Como llegamos a 3-5 clientes pagando en 30 dias?"*

### 4.1 Cold WhatsApp NO funciona como esta planteado

**Problema 1:** El numero en Google My Business es la linea del restaurante, NO el WhatsApp del dueno.

**Problema 2:** WhatsApp Business requiere opt-in. Mensajes no solicitados violan TOS. 5-10 reportes = baneo permanente del numero.

**Problema 3:** Conversion estimada: 100 mensajes frios -> 50 llegan -> 5 responden -> 1.5 demos -> 0.5 ventas. Necesitas 200 mensajes para UN cliente. Inviable en 30 dias con un solo developer.

**Propuesta: Invertir el flujo:**
1. Crear fotos AI mejoradas del restaurante objetivo
2. IR FISICAMENTE al restaurante (2-4pm, despues del rush)
3. Mostrar antes/despues en tablet
4. DESPUES de que te dan su WhatsApp personal, continuar por WhatsApp

### 4.2 Estrategia para los pilotos (La Unica + El Patron)

**Antes de que el producto exista:**
- Visitar ambos con camara, recopilar 50-100 fotos cada uno
- Entrevista de 30 min: entender dolores REALES
- Crear MANUALMENTE 4-5 posts para cada uno y publicar
- Documentar todo para caso de estudio

**Gratis o pago:** Primer mes gratis CON condiciones:
1. Permiso para caso de estudio
2. Video testimonial de 60 seg al final del mes
3. Pasan a $99/mes en mes 2
4. Deben responder WhatsApp en <24h para feedback

**Pedir referidos en semana 2**, cuando ya vieron 3-4 posts publicados. Un restaurantero satisfecho conoce 5-10 duenos mas.

### 4.3 Ranking de canales por velocidad (30 dias)

| # | Canal | Probabilidad | Resultado esperado | Costo |
|---|-------|-------------|-------------------|-------|
| 1 | Referidos de pilotos | 70% | 1-2 clientes | $0 |
| 2 | Puerta a puerta Charlotte | 60% | 1-2 clientes | Gasolina |
| 3 | Grupos Facebook locales | 40% | 0-1 cliente | $0 |
| 4 | Alianzas proveedores | 30% | 0-1 (medio plazo) | $20/referido |
| 5 | WhatsApp frio (como en PRD) | 15% | 0-1 | Riesgo baneo |

**Ejecutar canales 1, 2 y 3 simultaneamente.**

### 4.4 El funnel del PRD es para ESCALAR, no para los primeros 5

**Para los primeros 5 clientes, el "funnel" es:**
```
Dia 1: Contacto (visita o referido) -> mostrar fotos AI de SU restaurante
Dia 2-3: Enviar 2-3 posts de ejemplo por WhatsApp
Dia 3-5: Llamada/visita de cierre -> "$99/mes, sin contrato, 30 dias money-back"
Dia 5-7: Primer post real publicado
```

5-7 dias de ciclo, no 14. Venta directa, concierge, uno a uno.

### 4.5 Que hace que el primero diga SI

**Trigger #1 — Verguenza competitiva (el mas poderoso):**
"Mire el Instagram de [competidor cercano]. 3,000 seguidores, publica todos los dias. Su comida es mejor pero nadie lo sabe."

**Trigger #2 — Demo visual irresistible:**
SUS fotos, SU comida, SU local — viendose como restaurante de revista.

**Trigger #3 — Eliminacion de barrera:**
"Nosotros nos encargamos de TODO. Usted solo manda fotos por WhatsApp."

**El pitch de 60 segundos:**
> "Don Carlos, mire estas fotos de su restaurante [antes/despues]. Nosotros hacemos que sus redes se vean asi TODOS LOS DIAS. Solo me manda fotos por WhatsApp y nosotros creamos todo, lo publicamos, y le mandamos reporte cada semana. $99 al mes, sin contrato, y si no le gusta en el primer mes le devuelvo cada centavo. Empezamos esta semana?"

### 4.6 Split del tiempo 60/40

**60% construir (mananas) + 40% vender (tardes)**

| Semana | Foco Producto | Foco Ventas | Meta |
|--------|--------------|-------------|------|
| 1 | Setup tecnico | Onboarding pilotos gratis | 2 pilotos |
| 2 | Pipeline AI | Referidos + puerta a puerta | 2 + leads |
| 3 | WhatsApp flow | Cierre ventas | 3-4 (1-2 pago) |

### 4.7 Producto minimo para cobrar $99

1. Recibir fotos del dueno por WhatsApp (ya funciona)
2. Crear 3-4 posts/semana con AI (herramientas existentes)
3. Publicar en FB+IG (manualmente al inicio)
4. Reporte semanal por WhatsApp

4-5 horas/cliente/semana. Con 5 clientes = 20-25 hrs/semana. Laborioso pero viable mes 1-2.

# FASE 3 — ARCHITECTURE DECISION RECORD (ADR)

---

## ADR-001: Procesamiento Asincrono de Webhooks

**Decision:** Responder HTTP 200 inmediatamente. Guardar payload raw en `webhook_raw_log`. Procesar con worker loop cada 5 segundos.

**Justificacion:** GHL reintenta webhooks despues de 5s sin respuesta. Claude tarda 2-5s en clasificar. Sin procesamiento async, CADA mensaje genera duplicados.

**Alternativas descartadas:**
- Redis + BullMQ: Agrega dependencia innecesaria para MVP. La DB como cola es suficiente hasta 50 clientes.
- Procesar sincronicamente con timeout: No resuelve el problema de duplicados.

**Supuesto a validar:** GHL envia un campo unico por mensaje (messageId o similar) para idempotencia. Si no, generar hash del payload.

---

## ADR-002: Persistencia de Jobs en Base de Datos

**Decision:** Tabla `scheduled_jobs` con status (pending/processing/completed/failed). node-cron actua SOLO como poller cada minuto, no como scheduler.

**Justificacion:** node-cron es in-memory. Restart del servidor = jobs perdidos = clientes sin contenido. Con jobs en DB, al reiniciar el worker encuentra los pendientes y los ejecuta.

**Alternativas descartadas:**
- Cron externo (Railway cron, Render cron): Viable pero menos control sobre reintentos y status tracking.
- Agenda.js (MongoDB-backed scheduler): Agrega MongoDB como dependencia.

**Supuesto a validar:** Supabase puede manejar el polling cada minuto sin problemas de performance (deberia ser trivial con indices).

---

## ADR-003: Clasificacion de Intent Hibrida (Regex + Claude)

**Decision:** Regex primero para intents simples (aprobacion, rechazo, escalacion). Claude solo para intents complejos (FAQ, solicitud de contenido, multiples instrucciones).

**Justificacion:** 60-70% de mensajes son aprobaciones simples. Regex responde en <10ms vs Claude 2-5 segundos. Reduce costos y funciona como fallback si Claude cae.

**Alternativas descartadas:**
- Solo Claude para todo: Demasiado lento para aprobaciones, single point of failure.
- Solo regex: No puede manejar preguntas FAQ ni solicitudes complejas.

**Supuesto a validar:** La lista de variantes de "si" en espanol cubre 90%+ de aprobaciones reales. Validar con los pilotos.

---

## ADR-004: Transcripcion de Notas de Voz con Whisper

**Decision:** Integrar Whisper API (OpenAI) para transcribir notas de voz. Procesar la transcripcion con el mismo pipeline de texto.

**Justificacion:** Usuarios latinos envian 3x mas notas de voz que texto. Sin soporte de audio, 60%+ de comunicaciones se pierden. Costo: ~$0.003 por nota promedio.

**Alternativas descartadas:**
- Ignorar notas de voz en MVP: Inaceptable para el mercado latino. Es como hacer un producto sin soporte de texto.
- Google Speech-to-Text: Funciona pero Whisper tiene mejor soporte de espanol latinoamericano y Spanglish.

**Supuesto a validar:** Whisper transcribe correctamente espanol con ruido de cocina de fondo. Testear con grabaciones reales de los pilotos.

---

## ADR-005: Modo Diario Unico en MVP (Sin Modo Semanal)

**Decision:** Solo modo diario en MVP. Un post a la vez, enviado la noche anterior para aprobacion.

**Justificacion:** El modo semanal (6 posts de golpe) genera: sobrecarga de mensajes, confusion de cual post aprobar, aprobaciones parciales imposibles de parsear, y experiencia de spam. El modo diario es simple: 1 post, si o no.

**Alternativas descartadas:**
- Modo semanal con numeracion: Agrega complejidad significativa al parser de intents y a la UX.
- Ambos modos desde el inicio: Duplica el trabajo de desarrollo sin beneficio claro.

**Supuesto a validar:** Los clientes prefieren recibir 1 post/dia vs planificar toda la semana el lunes. Validar con pilotos.

## ADR-006: Onboarding en Micro-Sesiones

**Decision:** Onboarding dividido en 3-4 micro-sesiones de 5 minutos distribuidas en 3-4 dias, en lugar de una sesion unica de 60 minutos.

**Justificacion:** Un restaurantero trabajando 12-16 hrs/dia NO puede dar 60 minutos consecutivos de atencion en WhatsApp. Se va a interrumpir, perder contexto, y potencialmente abandonar.

**Alternativas descartadas:**
- Sesion unica de 60 min (como en PRD): Irreal para el target.
- Onboarding presencial: No escala, pero se puede hacer para pilotos.

**Supuesto a validar:** El cliente completa las 4 micro-sesiones en 4 dias. Si no, necesitamos recordatorios automaticos.

## ADR-007: Buffer de 15 Minutos Pre-Publicacion

**Decision:** Despues de aprobacion del cliente, esperar 15 minutos antes de publicar. Enviar mensaje: "Post aprobado. Se publicara en 15 min. Responde CANCELAR para detener."

**Justificacion:** Aprobaciones accidentales (manos con grasa, toque accidental) pueden publicar contenido con errores. El buffer da red de seguridad sin afectar operacion.

**Alternativas descartadas:**
- Publicacion inmediata: Riesgo de errores irreversibles.
- Confirmacion doble ("Estas seguro?"): Friccion excesiva para cada post.

## ADR-008: Database-as-Queue (Sin Redis en MVP)

**Decision:** Usar Supabase (PostgreSQL) como cola de mensajes con tablas `webhook_raw_log` y `scheduled_jobs`. Worker loops con setInterval.

**Justificacion:** Agregar Redis es una dependencia mas para mantener, deployar, y pagar. PostgreSQL con indices correctos maneja el volumen del MVP (< 500 eventos/dia) sin problema.

**Alternativas descartadas:**
- Redis + BullMQ: Overkill para MVP. Reservado para cuando el volumen lo justifique (50+ clientes).
- SQS/Cloud queues: Agrega dependencia de AWS.

**Punto de migracion:** Cuando el worker loop tome >10s por ciclo o cuando haya >1000 eventos/dia, migrar a BullMQ.

## ADR-009: Pilotos Gratis con Condiciones, Clientes 3-5 de Pago

**Decision:** La Unica y El Patron: primer mes gratis a cambio de caso de estudio + video testimonial. Clientes 3-5: $99/mes desde dia 1 con garantia de devolucion 30 dias.

**Justificacion:** El producto no existe aun. Cobrar a los pilotos por algo que se esta construyendo destruye confianza. Pero los clientes 3-5 DEBEN pagar para validar willingness-to-pay. La garantia de devolucion reduce riesgo sin devaluar.

## ADR-010: Venta Directa para Primeros 5 Clientes (Sin Funnel Automatizado)

**Decision:** Los primeros 5 clientes se consiguen con visita presencial, referidos de pilotos, y venta concierge. El funnel MIFGE -> Tripwire -> Bridge -> $99 se construye para escalar en mes 2-3.

**Justificacion:** El funnel completo tiene 12-15 touchpoints y toma 10-14 dias por lead. Con 30 dias de plazo y un developer construyendo el producto, no hay ancho de banda para operar un funnel automatizado. La venta directa convierte en 5-7 dias.

## TABLAS ADICIONALES AL SCHEMA (nuevas vs PRD original)

```sql
-- Contactos autorizados por restaurante (reemplaza phone_owner unico)
CREATE TABLE authorized_contacts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  restaurant_id UUID REFERENCES restaurants(id),
  phone TEXT NOT NULL,
  role TEXT DEFAULT 'owner',
  is_primary BOOLEAN DEFAULT FALSE,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(phone)
);

-- Estado de conversacion activa
CREATE TABLE conversation_state (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  restaurant_id UUID REFERENCES restaurants(id) UNIQUE,
  current_flow TEXT,
  flow_step INTEGER DEFAULT 0,
  flow_data JSONB DEFAULT '{}',
  expires_at TIMESTAMPTZ,
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- Jobs programados (persistentes, sobreviven restarts)
CREATE TABLE scheduled_jobs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  restaurant_id UUID REFERENCES restaurants(id),
  job_type TEXT NOT NULL,
  scheduled_for TIMESTAMPTZ NOT NULL,
  status TEXT DEFAULT 'pending',
  attempts INTEGER DEFAULT 0,
  last_error TEXT,
  completed_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_jobs_pending ON scheduled_jobs(scheduled_for) WHERE status = 'pending';

-- Log raw de webhooks (para idempotencia y dead letter queue)
CREATE TABLE webhook_raw_log (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  ghl_event_id TEXT UNIQUE,
  payload JSONB NOT NULL,
  status TEXT DEFAULT 'received',
  error_message TEXT,
  attempts INTEGER DEFAULT 0,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Subscripciones (tracking de pagos)
CREATE TABLE subscriptions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  restaurant_id UUID REFERENCES restaurants(id) UNIQUE,
  stripe_customer_id TEXT,
  stripe_subscription_id TEXT,
  status TEXT DEFAULT 'active',
  current_period_start TIMESTAMPTZ,
  current_period_end TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```
